Objects & State

How JavaScript stores and organizes your app's data

What is an Object?

Think of a JavaScript object as a **labeled filing cabinet**. Each drawer has a label (the key) and contains something (the value). In the Lyon app, everything from user settings to carpark information is stored in objects.

Here's how the Lyon app creates its main filing cabinet — the state object that holds everything:

CODE 
const state = {
    history: [],
    current: "carparks",
    userLocation: null,
    settings: {
        theme: "system",
        carparkSorting: "alphabetical"
    },
    pinnedCarparks: []
};
PLAIN ENGLISH

Create a permanent storage box called 'state'...

Keep track of where the user has been...

Remember which screen they're looking at right now...

User's location isn't known yet (set to null)...

Inside 'settings', create another filing cabinet...

Use the device's theme (light or dark mode)...

Sort carparks alphabetically by default...

Close the 'settings' cabinet...

Start with an empty list of favorite carparks.

💡
Why Objects Matter

Objects let you group related data together. Instead of having separate variables for theme and carparkSorting, they live inside state.settings — making it crystal clear they're both settings.

Accessing Object Properties

Once you have an object, you access its contents using dot notation. Think of it like giving directions: "Go to the state filing cabinet, open the settings drawer, and look at the theme."

📍

Reading Values

✏️

🪆

CODE 
function pushView(view) {
    if (view === state.current) return;
    state.history.push(state.current);
    state.current = view;
    render();
}

Nested Objects: Objects Inside Objects

The Lyon app uses nested objects to organize complex data. Look at how the travel planner is structured:

CODE 
planner: {
    plans: [],
    ui: {
        mode: "list",
        editingId: null
    }
}
PLAIN ENGLISH

The planner has its own cabinet...

With a list of all travel plans...

And a separate section for UI state...

This organization makes sense: state.planner.plans holds the actual plans, while state.planner.ui.mode tracks what the user interface is showing. It's like having a filing cabinet where one drawer contains documents (plans) and another drawer contains sticky notes about how you're organizing the documents (ui).

🎯
Real-World Pattern

Separating "data" from "UI state" is a professional pattern. The plans array is permanent data you'd save. The ui object is temporary state about what's displayed right now. When the app restarts, plans reload from storage but UI state resets to defaults.

Arrays & Array Methods

Working with lists of data like a pro

What is an Array?

An array is simply a numbered list. In the Lyon app, arrays store collections like pinned carparks, navigation history, and travel plans. Think of an array as a **playlist** — items are in a specific order, and you can add, remove, or rearrange them.

CODE 
const state = {
    history: [],
    pinnedCarparks: [],
    events: [],
    planner: {
        plans: []
    }
};
PLAIN ENGLISH

Create the main storage object...

History starts as an empty list (we'll add to it)...

No carparks are pinned yet...

Events list is empty until we fetch from the server...

Travel planner section...

With an empty list of plans...

💡
Arrays vs Objects

Array: Use when order matters or you have many similar items. Like a to-do list or a shopping cart.
Object: Use when you need labeled properties. Like a user profile with name, email, and age.

The Power Trio: map, filter, sort

JavaScript arrays have superpowers called array methods. The Lyon app uses three constantly: map (transform each item), filter (keep some, discard others), and sort (reorder items).

map: Transform Every Item

The map method takes a list and transforms each item. It's like having a factory assembly line — each item goes in, gets modified, and comes out the other side.

CODE 
let carparks = Object.entries(data).map(([name, details]) => ({
    name: name,
    availableSpaces: details.availableSpaces,
    latitude: details.latitude,
    longitude: details.longitude,
    isPinned: state.pinnedCarparks.includes(name)
}));
PLAIN ENGLISH

Take the raw carpark data and transform it...

Keep the carpark's name...

Keep how many spaces are available...

Keep its GPS coordinates...

(Another coordinate)...

Check if this carpark is in our pinned list...

Result: a cleaned-up list of carpark objects.

🎯
Why Transform Data?

The server sends raw data, but our app needs it in a specific shape. map lets us reshape every item in one go. We add isPinned to each carpark so the UI knows which ones to mark with a star.

filter: Keep Only What You Need

The filter method is like a bouncer at a club — it decides which items get through. The Lyon app uses it to separate pinned from unpinned carparks:

CODE 
const pinnedCarparks = carparks.filter(c => c.isPinned);
const unpinnedCarparks = carparks.filter(c => !c.isPinned);
PLAIN ENGLISH

Go through all carparks and keep only the UNpinned ones.

And here's filtering events by date range:

filteredEvents = state.events.filter(event => {

        return eventDate >= beginDate && eventDate <= endDate;

PLAIN ENGLISH

sort: Reorder the List

The sort method rearranges items. In Lyon, users can sort carparks alphabetically or by distance:

CODE 
if (state.settings.carparkSorting === "alphabetical") {
    unpinnedCarparks.sort((a, b) => a.name.localeCompare(b.name));
} else if (state.settings.carparkSorting === "distance") {
    unpinnedCarparks.sort((a, b) => a.distance - b.distance);
}
PLAIN ENGLISH

Check the user's sorting preference...

If they want alphabetical, compare names as text...

If they want nearest first...

Compare distances as numbers (smallest first).

The Spread Operator: Combining Arrays

The spread operator ... is like emptying a box onto a table. It unpacks an array's contents. The Lyon app uses it to combine pinned and unpinned carparks:

CODE 
const sortedCarparks = [...pinnedCarparks, ...unpinnedCarparks];
PLAIN ENGLISH

Create a new list by dumping all pinned carparks first, then all unpinned ones. Result: pinned appear at the top!

Without the spread operator, you'd have to write a loop to add each item. With spread, it's one line.

✨
Spread is Non-Destructive

[...pinnedCarparks, ...unpinnedCarparks] creates a new array and doesn't modify the originals. This is safer — you keep your original lists intact in case you need them again.

Chaining Array Methods

The magic happens when you chain methods together. Watch this elegant one-liner from the planner:

CODE 
function getSortedPlans() {
    return [...state.planner.plans].sort(planSorter);
}
PLAIN ENGLISH

When someone wants the plans in order...

Make a copy of the plans list (spread), then sort that copy by date and time. Return the sorted list.

The spread operator [...state.planner.plans] creates a copy first because sort modifies the original array. By copying first, we keep state.planner.plans in its original order and return a new sorted version.

DOM Manipulation

Making your web page come alive with JavaScript

What is the DOM?

The DOM is JavaScript's representation of your web page. Think of it as a **live puppet show** — the HTML is the script, and JavaScript pulls the strings to make elements move, appear, disappear, and change.

The Lyon app doesn't have separate HTML files for each screen. Instead, it has ONE empty container, and JavaScript fills it with content. Watch how the app grabs that container:

CODE 
const main = document.getElementById("mainContent");
const backBtn = document.getElementById("backBtn");
const viewTitle = document.getElementById("viewTitle");
const navButtons = document.querySelectorAll(".nav-btn");
PLAIN ENGLISH

Find the HTML element with id="mainContent" and store it...

Find the back button...

Find the title heading...

Find ALL buttons with the class "nav-btn" (Carparks, Events, etc).

🎯
Two Ways to Select Elements

getElementById finds ONE element by its id.
querySelectorAll finds ALL elements matching a CSS selector (like .nav-btn for a class).

Changing HTML Content

Once you've grabbed an element, you can change what's inside it using innerHTML. It's like clearing a whiteboard and writing something new.

The Lyon app uses innerHTML to completely rebuild the screen when switching views. Here's how it renders the carpark list:

Template Literals

Those backticks `...` create a template literal. The ${} syntax injects JavaScript values directly into the string. So ${focused.name} becomes "Lyon Center" or whatever the actual name is.

Fetch & Async/Await

Getting data from servers without freezing your app

The Problem: Waiting for Data

When your app needs data from a server (carpark availability, weather, events), it can't just ask and get an instant answer. The request travels across the internet, the server thinks, and sends a response back. This takes time — anywhere from milliseconds to seconds.

If JavaScript **waited** for the response (blocking), your entire app would freeze. You couldn't click buttons, scroll, or do anything until the data arrived. That's terrible UX.

🎯
The Asynchronous Solution

JavaScript uses asynchronous code. The app says "go fetch that data, and when it arrives, call me back." Meanwhile, the app stays responsive and can do other things.

fetch: Requesting Data

The fetch function sends requests to servers. Think of it as **ordering takeout** — you call the restaurant (send a request), they prepare your food (server processes), and eventually deliver it (response arrives).

Here's how the Lyon app fetches carpark data:

CODE 
async function renderCarparks() {
    const res = await fetch(API_BASE + "/carparks.json");
    const data = await res.json();
    // Now we have the data!
}
PLAIN ENGLISH

This function can pause and wait for things...

Send a request to the API for carpark data. Wait for the response...

Convert the response from JSON text to a JavaScript object. Wait for that to finish...

Comment explaining we're good to go!

Continue with the data...

💡
Two Magic Keywords

async in front of a function means "this function can pause and wait."
await means "pause here until this finishes, but don't freeze the app."

Understanding async/await

Let's break down what's happening step-by-step when Lyon fetches weather data:

CODE 
async function renderWeather() {
    const params = new URLSearchParams({
        lat: state.userLocation.latitude,
        lon: state.userLocation.longitude
    });
    const url = `${API_BASE}/weather.json?${params}`;
    const res = await fetch(url);
    const data = await res.json();
    // Display the weather...
}
PLAIN ENGLISH

This function fetches weather and can wait...

Create URL parameters with the user's GPS coordinates...

(Latitude value)...

(Longitude value)...

Done building params...

Build the full URL with the coordinates...

Send the request. PAUSE here until response arrives...

Parse the JSON. PAUSE until parsing finishes...

Now use the weather data!

1
Build the URL

Add user's location as query parameters

2
Send Request

await fetch(url) pauses until server responds

3
Parse JSON

4

Why Two Awaits?

You might wonder why there are two await statements. It's because fetch works in two stages:

1️⃣

First await

await fetch(url)
Waits for the server to respond with raw data

2️⃣

Second await

await res.json()
Waits for that raw data to be parsed into a JavaScript object

Think of it like mail delivery: first the package arrives (fetch completes), then you have to open it and unwrap what's inside (json() completes).

⚠️
Common Mistake

If you forget the second await, you'll get a Promise instead of the actual data. Always remember: fetch returns a response object, and you need to call .json() to extract the data.

Calling Async Functions

When you call an async function, you have two options:

Option 1: Await it (if you're inside another async function)

CODE 
async function render() {
    switch(state.current) {
        case "carparks":
            await renderCarparks();
            break;
    }
}
PLAIN ENGLISH

This render function is also async...

Check which view we're showing...

If it's carparks...

Wait for renderCarparks to finish fetching and rendering...

Then exit the switch...

Option 2: Use .then() for fire-and-forget

CODE 
getUserLocation().then(() => render());
PLAIN ENGLISH

Get the user's location, and when that's done, call render(). Don't wait — keep running other code.

Lyon uses .then() at the very end of the app to kick off the first render. It's like saying "once we know where the user is, show them the app."

Ensuring Dependencies are Ready

Sometimes you need data before you can fetch more data. The Lyon app checks if user location exists before fetching carparks (because it needs to calculate distances):

CODE 
async function renderCarparks() {
    const res = await fetch(API_BASE + "/carparks.json");
    const data = await res.json();
    
    if (!state.userLocation) {
        await getUserLocation();
    }
    // Now we have BOTH carpark data AND user location
}
PLAIN ENGLISH

When rendering carparks...

Fetch the carpark data from the server...

Parse it to JSON...

(Blank line for readability)

Check if we know where the user is...

If not, get their location and wait for it...

Done checking...

Now we can calculate distances!

This pattern ensures we never try to use data that doesn't exist yet. It's defensive programming — checking preconditions before proceeding.

localStorage

Remembering data between visits

The Memory Problem

By default, web apps have **amnesia**. When you close the tab or refresh the page, everything in JavaScript's memory disappears. State resets. Settings are lost. It's like your app starts from scratch every time.

This is a problem for the Lyon app. If a user pins their favorite carparks or changes to dark mode, they expect those choices to persist. Nobody wants to reconfigure their app every visit.

💾
Enter localStorage

localStorage is like a filing cabinet built into your browser. You can store small amounts of text data that survive page refreshes, browser restarts, and even system reboots.

Saving Data to localStorage

The Lyon app saves settings and pinned carparks using localStorage.setItem(). It takes two arguments: a key (like a file name) and a value (the content):

CODE 
function saveSettings() {
    localStorage.setItem("settings", JSON.stringify(state.settings));
}
PLAIN ENGLISH

When we need to save settings...

Convert the settings object to a text string, then save it to localStorage with the key "settings".

⚠️
localStorage Only Stores Text

localStorage can only save strings. So we use JSON.stringify() to convert JavaScript objects into text. The object {theme: "dark"} becomes the string '{"theme":"dark"}'.

Here's how pinned carparks are saved:

CODE 
function savePinnedCarparks() {
    localStorage.setItem("pinnedCarparks", JSON.stringify(state.pinnedCarparks));
}
PLAIN ENGLISH

When saving pinned carparks...

Convert the array to a text string and save it with the key "pinnedCarparks".

Loading Data from localStorage

When the app starts, it needs to check if there's saved data and restore it. The Lyon app does this with localStorage.getItem():

CODE 
function loadSettings() {
    const saved = localStorage.getItem("settings");
    if (saved) {
        state.settings = JSON.parse(saved);
    }
    const pinnedCarparks = localStorage.getItem("pinnedCarparks");
    if (pinnedCarparks) {
        state.pinnedCarparks = JSON.parse(pinnedCarparks);
    }
}
PLAIN ENGLISH

When the app first loads...

Check if there's saved settings in localStorage...

If something was saved...

Convert the text string back to a JavaScript object and restore it...

Done with settings...

Check if there are pinned carparks saved...

If there are...

Convert the string back to an array and restore it...

Now state is restored from last visit!

🔄
The Round Trip

Saving: Object → JSON.stringify() → String → localStorage.setItem()
Loading: localStorage.getItem() → String → JSON.parse() → Object

When to Save vs When to Load

The Lyon app follows a clear pattern:

📥
Load on Startup

loadSettings() and loadPlanner() run immediately when the app starts, before rendering anything.

💾
Save on Change

Whenever the user changes a setting or pins a carpark, the app calls the save function immediately.

Look at how the theme setting works:

    input.addEventListener("change", (e) => {

For each theme radio button...

What NOT to Store

localStorage has limits. The Lyon app is smart about what it persists:

✅

DO Store

User preferences (theme, sort order)
User-created data (pinned items, travel plans)
Small amounts of text

❌

DON'T Store

Data from APIs (refetch it)
Temporary UI state (current screen)
Sensitive data (passwords)

📊
Why Not Store API Data?

Carpark availability and event listings change frequently. If Lyon saved them to localStorage, users would see stale data. Better to fetch fresh data each visit and only save user preferences.

Advanced Patterns

Modern JavaScript tricks that make code cleaner and safer

Destructuring: Unpacking Objects & Arrays

Destructuring is like opening a gift box and taking out specific items. Instead of reaching into an object repeatedly, you extract what you need upfront.

The Lyon app uses destructuring when transforming carpark data:

CODE 
let carparks = Object.entries(data).map(([name, details]) => ({
    name: name,
    availableSpaces: details.availableSpaces,
    latitude: details.latitude,
    longitude: details.longitude
}));
PLAIN ENGLISH

For each carpark entry (which is [name, detailsObject])...

Unpack it into two variables: 'name' and 'details'...

Keep the name...

Extract available spaces from the details object...

Extract latitude...

Extract longitude...

Create a new object with this cleaned-up data.

✨
Array Destructuring

[name, details] unpacks an array. Object.entries() returns arrays like ["Lyon Center", {...}], and destructuring splits them into separate variables.

Here's object destructuring in the planner UI:

CODE 
const {mode, editingId} = state.planner.ui;
PLAIN ENGLISH

Open the state.planner.ui object and pull out 'mode' and 'editingId' as separate variables. Now we can use them directly!

Without destructuring, you'd write:

const mode = state.planner.ui.mode;
const editingId = state.planner.ui.editingId;

Destructuring does the same thing in one line!

The Spread Operator (Again, But Deeper)

We saw the spread operator with arrays. It also works with objects and has powerful uses:

Combining Sorted Arrays

CODE 
const sortedCarparks = [...pinnedCarparks, ...unpinnedCarparks];
PLAIN ENGLISH

Create a new array by spreading out all pinned carparks first, then all unpinned ones. Result: one merged list!

Copying Before Sorting

CODE 
function getSortedPlans() {
    return [...state.planner.plans].sort(planSorter);
}
PLAIN ENGLISH

When someone wants sorted plans...

Make a shallow copy of the plans array, then sort THAT copy. This protects the original from being modified.

🛡️
Why Copy First?

sort() modifies the array in place. By spreading first ([...array]), we create a copy, so the original state.planner.plans stays in its insertion order. Different parts of the app can sort differently without conflicts.

Optional Chaining: Safe Navigation

Optional chaining (?.) prevents crashes when accessing nested properties that might not exist. It's like asking "if this exists, give me the next thing; otherwise, just give me undefined."

The Lyon app uses it when displaying plan images:

CODE 
${editingPlan?.image ? `<img src="${editingPlan.image}">` : ''}
PLAIN ENGLISH

If editingPlan exists AND it has an image, show the image tag. Otherwise, show nothing. No crash if editingPlan is null!

Without optional chaining, you'd have to write:

if (editingPlan && editingPlan.image) {
    // show image
}

Optional chaining collapses that into one clean expression.

⚡
When to Use It

Use optional chaining when accessing properties that might not exist — user uploads, API responses with optional fields, or when an object might be null. It makes code safer and more concise.

matchMedia: Detecting User Preferences

matchMedia lets JavaScript ask "what are the user's system preferences?" The Lyon app uses it to detect if the user has dark mode enabled:

CODE 
function applyTheme() {
    const theme = state.settings.theme;
    
    if (theme === "system") {
        const prefersDark = window.matchMedia("(prefers-color-scheme: dark)").matches;
        if (prefersDark) {
            document.body.classList.add("dark-theme");
        } else {
            document.body.classList.add("light-theme");
        }
    }
}

Respecting User Choice

matchMedia checks the user's OS-level preference. If they've set their phone or computer to dark mode, .matches will be true. This makes the app feel native and respectful of their preferences.

You can also use matchMedia for responsive design:

const isMobile = window.matchMedia("(max-width: 768px)").matches;
if (isMobile) {
    // Show mobile layout
}

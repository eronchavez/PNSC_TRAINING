Lyon Mobile Web Service: Development Guide
Development Guide

Gagawa tayo ng Lyon Mobile Web Service mula umpisa, step by step.

Ang goal ng course na ito ay hindi lang ipakita kung ano ang nasa code ngayon. Ang gusto natin ay maintindihan mo kung paano mo siya bubuuin as a beginner: mula sa app shell, hanggang state, navigation, API views, planner CRUD, settings, local storage, at PWA setup.

Isipin mo na nasa classroom tayo. Ako ang instructor mo, at ang target natin ay hindi memorization kundi malinaw na mental model. Sa bawat part, sasagutin natin ang tanong na: "Bakit ito kailangan, at saan ito dapat ilagay sa code?"

App Shell

Nandito ang header, back button, bottom navigation, at ang isang main container kung saan ipapakita ang kasalukuyang view.

01

Foundation: Unang Itatayo

Bago ka maglagay ng features, kailangan malinaw muna ang shell ng application.

Step 1: Ayusin muna ang file structure

Dito nagsisimula ang shell. Sa index.html mo ilalagay ang fixed header, main content area, at bottom nav. Huwag mo munang isipin ang API or planner. Scaffold muna bago feature.

Dito titira ang logic. Sa js/app.js mo ilalagay ang state, render functions, event listeners, at boot sequence. Ito ang magiging utak ng app.

Dito naman ang presentation. Sa css/main.css mo i-aayos ang mobile layout, cards, theme colors, spacing, at scroll behavior.

Optional sa simula, pero magandang ihanda agad. Ang manifest.json ang nagsasabi sa browser na puwedeng i-install ang app sa phone home screen.

02

State, Router, at Shared Rendering

Ngayong may shell na, tuturuan natin ang app kung paano lumipat ng screens.

Step 5: Gumawa ng title map at navigation state

CODE 
const titleMap = {
  carparks: "Carparks",
  events: "Events",
  weather: "Weather",
  planner: "Travel Planner",
  settings: "Settings"
};

function setHeader(view) {
  viewTitle.textContent = titleMap[view] ?? "Carparks";
  backBtn.disabled = state.history.length === 0;
}
EXPLANATION

Simple lang pero important: hindi hardcoded ang title sa header. Nakabase ito sa current view.

Sa beginner projects, malaking tulong ang ganitong mapping para hindi paulit-ulit ang strings sa iba-ibang functions.

Nadadamay rin dito ang back button state, kaya isang place lang ang inaalagaan mo.

Step 6: Gumawa ng mini router gamit lang ang state

CODE 
function pushView(view) {
  if (view === state.current) return;
  state.history.push(state.current);
  state.current = view;
  render();
}

function goBack() {
  if (state.history.length === 0) return;
  const previousView = state.history.pop();
  state.current = previousView;
  render();
}
EXPLANATION

Ito ang pinakasimpleng router. Wala tayong framework dito, plain JavaScript lang.

Kapag pinindot ang nav button, isi-save muna natin ang lumang view sa history, saka papalitan ang current.

Pag pinindot ang back, babalikan natin ang last view sa stack. Very practical ito para maintindihan ng beginners ang navigation flow.

Step 7: Gumawa ng main render dispatcher

CODE 
async function render() {
  setActiveNav(state.current);
  setHeader(state.current);

  switch (state.current) {
    case "carparks":
      renderCarparks();
      break;
    case "events":
      renderEvents();
      break;
    case "weather":
      renderWeather();
      break;
    case "planner":
      renderPlanner();
      break;
    case "settings":
      renderSettings();
      break;
  }
}
EXPLANATION

Ito ang traffic enforcer ng app. Siya ang nagsasabing: "Base sa current state, aling screen ang dapat ipakita?"

Kapag gusto mong magdagdag ng bagong tab, dito mo siya karaniwang isasaksak.

Importanteng maintindihan ito ng beginner dahil dito umiikot ang buong architecture ng app.

03

Pagbuo ng Mga Feature Views

Dito na papasok ang tunay na laman ng app: carparks, events, weather, at travel planner.

Feature map: hindi pare-pareho ang style ng bawat view

Carparks ang pinaka-magandang simulan sa feature layer. May fetch, may data transformation, may sorting, may pinning, at may focus mode pa.

Events ay magandang example ng filterable list. Kumuha ka ng data, lagyan ng inputs, tapos i-filter mo gamit ang state.

Weather ang example ng data + asset composition. Hindi lang JSON ang kinukuha mo, pati matching SVG per weather status.

Planner ang mini app sa loob ng app. May sariling UI mode, sariling validation, sariling save/load flow, at local CRUD behavior.

Ito ang complete thinking process ng carparks feature.

Bakit mahalaga ang derived data sa carparks?

Ang distance ng bawat carpark ay hindi naka-save sa localStorage at hindi rin kasama sa raw API response bilang final answer. Kino-compute natin ito mula sa latitude at longitude ng user at ng carpark. Ibig sabihin, derived data siya.

TIP
Instructor note

Kapag may student na gustong magdagdag ng "nearest first" feature, ang tamang sagot ay hindi "mag-save ng distance sa localStorage." Ang tamang sagot ay "i-compute muli ang distance habang bina-build ang view."

Step 9: Events at Weather bilang dalawang ibang pattern

E

Events: fetch tapos filter

I-save mo muna ang fetched events sa state, tapos gumamit ng date inputs para i-filter ang list every time na may change ang user.

W

Weather: fetch tapos compose

Bukod sa JSON, kailangan mo ring i-fetch ang tamang SVG icon per status. Magandang example ito ng pag-combine ng data at assets.

Step 10: Planner ang huling major feature dahil mas mabigat ang logic

CODE 
if (mode === "create" || mode === "edit") {
  main.innerHTML = buildPlannerForm();
} else {
  main.innerHTML = buildPlannerList();
}

plannerForm.addEventListener("submit", async (e) => {
  e.preventDefault();
  const formData = new FormData(plannerForm);
  const imageFile = formData.get("imageFile");
  if (imageFile && imageFile.size > 0) {
    imageValue = await fileToBase64(imageFile);
  }
  savePlanner();
  renderPlanner();
});
EXPLANATION

Ang planner ay hindi simpleng list lang. May dalawang malaking mode siya: form mode at list mode.

Sa create/edit mode, kailangan mong mag-validate ng inputs, basahin ang file gamit ang FormData, at i-convert ang image to base64.

Pagkatapos nito, ise-save mo ang plans, isosort mo sila, at ibabalik ang UI sa list mode.

Recommended build order para hindi ka malito

1
Carparks muna

Makukuha mo rito ang fetch, rendering, sorting, event handling, at location logic sa isang feature pa lang.

2
Events at Weather next

Mas madali na sila pagkatapos ng carparks kasi variation na lang sila ng fetch-and-render pattern.

3
Planner last

Mas maraming moving parts ang planner: form state, edit mode, delete, sorting, image handling, at persistence.

04

Persistence, Settings, at PWA Finish

Kapag gumagana na ang features, saka mo palalakasin ang memory at mobile-app feel ng project.

Step 11: I-save ang user preferences at planner data

S

Settings

Theme at carpark sorting ang ise-save mo sa settings key para bumalik ang preferred behavior kapag ni-refresh ang page.

P

Pinned Carparks

Dahil user choice ito, hindi puwedeng mawala after reload. Kaya gumagamit tayo ng hiwalay na key para sa pinned names.

T

Travel Planner

Plans at planner UI mode ay parehong local-first. Hindi kailangan ng backend para sa basic persistence nito.

R

Restore on boot

Saving is only half the story. Dapat may matching load functions ka rin sa startup sequence para bumalik ang data sa state.

Step 13: PWA layer para maging installable

PWA
Hindi ito cosmetic lang

Kapag may manifest.json, theme color, touch icon, at mobile web app meta tags ka, nagiging mas app-like ang web project. Puwede itong lumabas sa home screen at mag-open in standalone mode depende sa platform.

Bilang beginner, magandang tandaan ito: ang PWA ay outer packaging. Hindi nito pinapalitan ang state, render, o feature logic. Pinapaganda lang nito ang delivery ng app sa mobile device.

Final summary: paano mo ito bubuuin sa tamang order

1
Scaffold

Gawin muna ang HTML shell, CSS layout, shared header, at bottom nav.

2
State + Router

Gumawa ng central state object, header helpers, nav listeners, back logic, at render dispatcher.

3
Core Features

Idagdag ang carparks, events, weather, at planner one by one, starting from the simplest repeatable pattern.

4
Persistence + Polish

I-save ang settings at plans, i-restore sa startup, at saka idagdag ang PWA manifest at mobile polish.
